# 测试规约

> 定义按项目阶段差异化的测试策略、执行标准和 G5 Gate 检查清单。
> 测试层级定义和分级门槛见 `sdlc-governance.md` §8。
> Phase 在 Project Profile 中声明，影响测试策略选择。

---

## 1. 项目阶段定义 (Phase)

| Phase | 名称 | 判定标准 | 测试核心关注 |
|-------|------|----------|-------------|
| P1 | 新建 (Bootstrap) | 从零开始，无历史代码，无已有测试 | 核心路径覆盖、设计意图验证、测试基础设施搭建 |
| P2 | 成长 (Growth) | 已有基础代码，功能快速新增中 | 覆盖率跟上增长速度、接口契约稳定、CI 集成 |
| P3 | 成熟迭代 (Stable) | 功能基本稳定，以迭代优化和维护为主 | 回归保护、增量覆盖、影响分析、测试债务治理 |
| P4 | 遗留/接管 (Legacy) | 接手他人代码，或系统老化缺少测试 | 建立安全网、characterization test、逐步补覆盖 |

### 1.1 阶段判定规则

- Phase 在 Project Profile 中显式声明（`project_phase: P1/P2/P3/P4`）
- 如未声明，Agent 根据以下信号推断：
  - 项目无 test 目录或测试文件数 = 0 → P1 或 P4
  - 项目有测试但覆盖率 < 30% → P2 或 P4
  - 项目有成熟测试体系（CI 通过、覆盖率 > 60%）→ P3
- P1 vs P4 的区分：是否存在大量未测试的已有代码
- 不确定时默认：**P2**

### 1.2 阶段可演进

项目阶段不是永久的。典型演进路径：

```
P1 (新建) → P2 (成长) → P3 (成熟迭代)
P4 (遗留接管) → P2 (成长) → P3 (成熟迭代)
```

当项目特征发生变化时（如覆盖率达标、CI 稳定），应在 Project Profile 中更新 Phase。

## 2. 通用测试原则（所有阶段适用）

### 2.1 测试层级（引用 sdlc-governance §8）

| Level | 范围 | 验证目标 |
|-------|------|----------|
| L1 单元 | 单函数/模块 | 逻辑正确性 |
| L2 集成 | 跨模块、API 契约、DB | 模块协作正确性 |
| L3 系统/E2E | 完整用户路径 | 端到端行为正确性 |
| L4 验收 | 设计意图 | 需求满足度 |

### 2.2 测试代码基本规范

- **命名**：测试函数名应描述被测行为和预期结果，如 `test_login_with_expired_token_returns_401`
- **结构**：遵循 Arrange-Act-Assert（或 Given-When-Then）模式
- **断言**：每个测试聚焦一个行为，断言应具体（避免 `assertTrue(result)`）
- **独立性**：测试之间不应有执行顺序依赖
- **可重复性**：相同输入必须产出相同结果（除非测试非确定性系统，见 §7.3）

### 2.3 禁止模式

| 禁止模式 | 说明 | 替代做法 |
|----------|------|----------|
| 硬编码通过 | `assert True` 或注释掉失败断言 | 修复代码或标记为 `skip` 并记录原因 |
| 忽略失败 | `try/except pass` 吞掉测试异常 | 让测试失败，分析根因 |
| 过度 mock | mock 到只测试 mock 本身而非真实逻辑 | mock 外部边界，不 mock 被测逻辑 |
| 假绿测试 | 测试通过但不验证任何有意义的行为 | 确保断言覆盖核心行为 |
| 环境泄漏 | 测试修改全局状态影响其他测试 | 用 setUp/tearDown 隔离 |

## 3. P1 新建阶段测试策略

### 3.1 优先级

```
核心路径覆盖 > 边界条件 > 异常场景 > 性能基线
```

先确保"主路径能跑通"，再逐步扩展边界和异常。

### 3.2 测试基础设施搭建清单

新项目在编写第一个功能前，应先完成：

```
□ 选定测试框架（与项目语言/框架匹配）
□ 创建测试目录结构（如 tests/unit/, tests/integration/）
□ 编写第一个 smoke test（验证项目能启动）
□ 配置 CI 管道运行测试（如有）
□ 在 Project Profile 中记录测试命令
```

### 3.3 Mock 策略

- 外部依赖（第三方 API、数据库）未稳定时，可使用 mock
- 所有 mock 点必须标记为 `# TODO: replace mock with real dependency when stable`
- mock 应模拟真实行为（包括错误场景），不是总返回成功

### 3.4 覆盖率目标

| 范围 | 目标 |
|------|------|
| 核心模块 | >= 70% 行覆盖 |
| 非核心模块 | 无硬性要求，但核心路径必须有 |
| 总体 | 不设硬性总体目标 |

### 3.5 G5 检查清单（P1）

```
□ 核心路径有 L1 测试覆盖
□ 至少 1 个 L2 集成测试验证模块间协作
□ 测试全部通过（无 skip 除非有记录原因）
□ 测试基础设施已搭建（框架、目录、CI）
□ 无硬编码通过或假绿测试
```

## 4. P2 成长阶段测试策略

### 4.1 优先级

```
新增代码覆盖 > 接口契约稳定 > 集成验证 > 回归防护
```

### 4.2 增量覆盖率规则

- **新增代码**：每个 PR/任务的新增代码覆盖率 >= 80%
- **修改代码**：修改的函数必须有对应测试（如原来没有，需补上）
- **总体趋势**：总覆盖率应随开发递增，不允许大幅下降

### 4.3 接口契约测试

- 新增或变更 API 端点必须有对应的契约测试
- 模块间接口变更必须更新相关集成测试
- 数据库 schema 变更必须有迁移测试

### 4.4 Mock 过渡

- 随着依赖稳定，逐步将 mock 替换为真实依赖测试
- 每个迭代至少替换 1-2 个 mock 点
- 保留 mock 用于 CI 速度优化，但定期运行真实依赖测试

### 4.5 G5 检查清单（P2）

```
□ 新增代码覆盖率 >= 80%
□ 修改的函数有对应测试
□ 新增/变更 API 有契约测试
□ L1 + L2 测试通过
□ 至少 1 条 L3 路径覆盖主流程（T2+ 任务）
□ 无新增假绿测试
□ CI 管道通过（如已配置）
```

---

## 5. P3 成熟迭代阶段测试策略

### 5.1 优先级

```
回归保护 > 影响分析 > 增量覆盖 > 测试债务治理
```

### 5.2 回归测试策略

每次变更执行影响分析：

```
1. 识别变更文件
2. 分析依赖关系（谁调用了被改的代码）
3. 确定必跑测试集（直接相关 + 间接影响）
4. 全量运行必跑测试集
5. 如影响范围不明确 → 运行全量测试
```

### 5.3 增量覆盖率

| 指标 | 要求 |
|------|------|
| 新增代码覆盖率 | >= 80% |
| 修改代码覆盖率 | >= 80% |
| 总体覆盖率 | 不允许下降（基线为当前值） |
| 核心模块覆盖率 | >= 80%（T2+任务） |

### 5.4 测试债务识别

定期（建议每月或每 Sprint）检查以下测试债务：

| 债务类型 | 检测方式 | 治理方式 |
|----------|----------|----------|
| 过时测试 | 测试的被测代码已删除或重写 | 删除或重写测试 |
| 假绿测试 | 测试通过但不验证有意义行为 | 加强断言 |
| 覆盖盲区 | 核心模块覆盖率 < 60% | 补充测试 |
| 脆弱测试 | 非确定性失败（flaky test） | 修复或隔离 |

### 5.5 G5 检查清单（P3）

```
□ 影响分析已完成，必跑测试集已识别
□ 必跑测试集全部通过
□ 新增/修改代码覆盖率 >= 80%
□ 总体覆盖率未下降
□ L1 + L2 + L3 测试通过（T2+ 任务）
□ 无新增脆弱测试
□ 回归测试未发现新问题
□ 如发现测试债务 → 已记录到 backlog
```

## 6. P4 遗留/接管阶段测试策略

### 6.1 优先级

```
安全网（要动的部分）> 关键路径 > 逐步补覆盖
```

**核心原则**：不改的代码不需要立即补测试，但要改的代码必须先有测试再改。

### 6.2 Characterization Test

在修改遗留代码前，先记录现有行为：

```
1. 对目标函数/模块编写 characterization test
2. 运行测试，用实际输出作为"预期值"
3. 确认测试通过（记录的是当前行为，不是"正确行为"）
4. 然后再进行修改
5. 修改后：如预期行为变化 → 更新测试；如不应变化 → 测试应仍通过
```

### 6.3 安全网策略

```
要改模块 A → 先为 A 补 L1 测试 → 确认测试通过 → 再改 A → 验证测试仍通过
```

- 补测试的范围仅限"要动的部分"，不要求全项目补齐
- 每次修改逐步扩大安全网覆盖范围

### 6.4 覆盖率目标

| 范围 | 目标 |
|------|------|
| 本次修改涉及的模块 | >= 60% |
| 全项目 | 不设硬性目标 |
| 趋势 | 每次变更后覆盖率应递增 |

### 6.5 G5 检查清单（P4）

```
□ 修改涉及的模块有 characterization test 或已有测试
□ 修改后测试通过（行为未意外改变）
□ 如有意改变行为 → 测试已同步更新
□ 安全网覆盖了本次变更的核心路径
□ 无"改了代码但没跑相关测试"的情况
□ 已记录本次新增的测试覆盖范围（供后续参考）
```

## 7. 项目类型特殊测试要求

除阶段策略外，不同项目类型有额外的测试关注点：

### 7.1 Type A — 脚本/自动化

| 关注点 | 测试方法 |
|--------|----------|
| 幂等性 | 相同输入执行两次，结果一致 |
| 重试安全 | 中途失败后重新执行不产生副作用 |
| 输入边界 | 空输入、超大输入、格式错误的输入 |
| 退出码 | 成功和失败场景的退出码正确 |

### 7.2 Type B — App/API/网站

| 关注点 | 测试方法 |
|--------|----------|
| 端点覆盖 | 每个 API 端点至少有 happy path + 1 个错误场景 |
| 错误码 | 4xx/5xx 错误码和错误消息正确 |
| 认证/授权 | 未授权访问被正确拒绝 |
| 并发安全 | 关键操作的并发场景（如适用） |
| 数据验证 | 输入校验拒绝非法数据 |

### 7.3 Type C — AI Agent/工作流

AI 系统的输出具有非确定性，传统的精确断言不完全适用：

| 关注点 | 测试方法 |
|--------|----------|
| 行为约束 | 验证 Agent 没做被禁止的事（如未修改不该改的文件） |
| 输出范围 | 结果在合理范围内（如"输出长度 < 1000 字"） |
| 格式遵从 | 输出格式符合要求（JSON 合法、字段完整） |
| Prompt 回归 | prompt 变更后，用标准用例验证行为未偏离 |
| 状态闭合 | 工作流正确终止，无悬挂状态 |
| 假完成检测 | 验证 Agent 声称的"完成"是否真的完成 |

**Prompt 回归测试方法**：
```
1. 维护一组标准测试用例（输入 + 预期行为描述）
2. 每次 prompt 变更后运行全部标准用例
3. 人工或 LLM 判定输出是否满足预期行为描述
4. 如有偏离 → 记录为回归问题
```

### 7.4 Type D — 集成/数据流

| 关注点 | 测试方法 |
|--------|----------|
| 契约测试 | 上下游接口契约一致（schema、字段、类型） |
| 超时处理 | 外部调用超时时正确降级或重试 |
| 重试安全 | 重试不产生重复数据 |
| 去重验证 | 相同消息多次投递结果一致 |
| 断连恢复 | 连接断开后能自动恢复 |

## 8. G5 Gate 检查矩阵

按 Gate Profile x Phase 交叉定义 G5 reviewer 的检查重点：

### 8.1 G-Lite (T1)

| Phase | G5 检查重点 |
|-------|------------|
| P1 | 核心路径有 L1 测试，测试能跑通 |
| P2 | 新增代码有 L1 测试 |
| P3 | 修改相关的 L1 测试通过 |
| P4 | 被改模块有 characterization test |

### 8.2 G-Std (T2)

| Phase | G5 检查重点 |
|-------|------------|
| P1 | L1 + L2 覆盖核心路径，至少 1 条 L3 路径，测试基础设施就绪 |
| P2 | 新增代码覆盖率 >= 80%，接口契约测试更新，L1 + L2 + L3 |
| P3 | 影响分析完成，回归测试通过，覆盖率未下降，L1 + L2 + L3 |
| P4 | 安全网覆盖变更范围，characterization test 通过，L1 + L2 |

### 8.3 G-Full (T3)

| Phase | G5 检查重点 |
|-------|------------|
| P1 | L1 + L2 + 完整 L3 + L4 验收，核心模块覆盖 >= 70% |
| P2 | 全部 G-Std 要求 + L4 验收 + 增量覆盖率 >= 80% |
| P3 | 全部 G-Std 要求 + L4 验收 + 回归全量通过 + 测试债务已记录 |
| P4 | 全部 G-Std 要求 + L4 验收 + 安全网策略文档化 |
