# AgentSDLC 治理规约 — 通用版

> 基于 AgentSDLC v3.1，适配 AI Agent（controller + implementer）双角色工作模式。
> 本规约为通用版本，各 agent 适配层可覆盖或补充。

---

## 1. 角色定义

在 Agent 工作环境中：
- **Host Agent = controller + 主实施者**（直接编码、测试、review）
- **Claude Worker = 可选外部独立 reviewer**（通过 CLI 调用 Claude Code）
- Host Agent 在 T1/T2 日常任务中兼任 controller 和 implementer
- 在 T2+ 高风险和 T3 任务中，Host Agent 必须通过 role-switching 或调用 Claude Worker 实现独立审查

---

## 2. 任务分类体系

**在开始任何非 trivial 任务前，必须先完成分类。**

### 2.1 项目等级 (Level)

| Level | 名称 | 示例 |
|-------|------|------|
| L1 | 一次性/低持久 | 临时脚本、demo、一次性数据任务 |
| L2 | 持久个人项目 | 个人 API、长期运行 agent、私有系统 |
| L3 | 多用户/高损失 | 外部服务、共享系统、有删除能力的系统 |

不确定时默认：**L2**

### 2.2 项目类型 (Type)

| Type | 名称 | 主要关注 |
|------|------|----------|
| A | 脚本/自动化 | 输入输出正确性、幂等、重试安全 |
| B | App/API/网站 | 主路径可用、兼容性、错误处理 |
| C | AI agent/工作流 | 上下文正确性、状态闭合、假完成 |
| D | 集成/数据流 | 协议稳定、超时重试、去重处理 |

### 2.3 变更风险 (Risk)

| Risk | 名称 | 示例 |
|------|------|------|
| R1 | 低风险 | 文档、注释、UI 微调、非行为性清理 |
| R2 | 中风险 | 标准功能、局部逻辑变更、prompt 调优 |
| R3 | 高风险 | 状态/内存逻辑、schema 迁移、核心循环、权限 |

不确定时默认：**R2**

**自动升级到 R3** 的触发条件：
- 涉及 memory 或长期状态
- 状态机或状态转换规则
- 任务编排或调度器逻辑
- 持久化布局或迁移
- 删除路径或权限逻辑
- 核心外部连接器
- 主 agent 循环或控制循环
- 生产事故修复

### 2.4 变更类别 (Class)

| Class | 名称 | 典型场景 |
|-------|------|----------|
| A | 安全本地变更 | 小 bugfix、文档修正、窄测试添加 |
| B | 运行时行为变更 | 新路由、API 变更、工作流分支、状态机 |
| C | 环境/数据/schema/memory | DB 迁移、memory 契约、配置、知识接线 |
| D | 自演化/边界/权限 | 自动修改规则、review 绕过、变异权限 |

**Class 覆盖规则**：
- Class B → 持久项目最低 T2
- Class C → 最低 T2；L2/L3 默认 T3
- Class D → **强制 T3 + R3**，禁止自接受，禁止直接上线，**需人类签批**

不确定时默认：**Class B**

### 2.5 项目阶段 (Phase)

| Phase | 名称 | 判定标准 |
|-------|------|----------|
| P1 | 新建 (Bootstrap) | 从零开始，无历史代码，无已有测试 |
| P2 | 成长 (Growth) | 已有基础代码，功能快速新增中 |
| P3 | 成熟迭代 (Stable) | 功能基本稳定，以迭代优化和维护为主 |
| P4 | 遗留/接管 (Legacy) | 接手他人代码，或系统老化缺少测试 |

Phase 在 Project Profile 中声明，影响测试策略选择（详见 `testing-standards.md`）。
Phase 不影响 Level/Type/Risk/Class 的分类流程。

不确定时默认：**P2**

### 2.6 治理模板与 Gate Profile

| Template | 名称 | 适用 | Gate Profile |
|----------|------|------|-------------|
| T1 | 轻量级 | L1+R1/R2, 或 L2+R1 | G-Lite |
| T2 | 标准 | L2+R2, L1+R3, 大多数持久工作 | G-Std |
| T3 | 加强 | L2+R3, L3+任意, 爆炸半径大 | G-Full |

**所有 7 个 Gate（G1-G7）始终存在。** Gate Profile 决定每个 Gate 的严格程度，而非是否存在。

### 2.7 分类输出格式

```
## Task Classification
- Task ID:
- Project Level: L1 / L2 / L3
- Project Type: A / B / C / D
- Project Phase: P1 / P2 / P3 / P4
- Change Risk: R1 / R2 / R3
- Change Class: A / B / C / D
- Governance Template: T1 / T2 / T3
- Gate Profile: G-Lite / G-Std / G-Full

## Why
- <分类理由>
- <是否应用了 class override>
```

## 3. 不可协商规则

### 3.1 禁止自接受
Agent 不可接受自己的输出为最终结果。
- T1: 必须进行显式 role-switching（implementer → reviewer 视角切换）
- T2+: 必须产出 review 结果（pass / conditional pass / reject）
- T3 / Class C/D: 必须调用 Claude Worker 或其他外部独立审查

### 3.2 不确定时停止并报告
如果数据、API、schema 或上下文不足：
- 停止执行
- 向用户报告缺口
- 禁止发明行为

### 3.3 有界工作
一个任务、一个范围、一个结果。
如果范围在执行中扩展，停止并报告。

### 3.4 证据优于断言
不得在没有证据的情况下声称"已修复""已验证""可发布""安全"。

### 3.5 设计先行
非 trivial 任务不得直接跳入编码。每个任务需要至少与其治理层级匹配的最小设计。

### 3.6 回滚始终必需
每次发布必须回答：如何发现故障、如何止损、如何回滚。

### 3.7 管道纪律
- 实施/review/验证中不超过 3 个活跃任务
- 晋升中不超过 1 个活跃任务
- 管道过载时不启动新的主要工作（除非处理事故）

### 3.8 Class D 需人类签批
自演化变更需人类审批。Controller 准备草案，人类批准。

### 3.9 闭环反馈
运行时信号创建新的已分类任务，而非热编辑。

## 4. 七阶段生命周期与 Gate

```
Design → Design Review → Implementation → Code Review → Testing → Go-to-Production → Monitoring
  ^                                                                                      |
  |______________________________________________________________________________________|
                                       (Feedback Loop)
```

### Gate 速查表

| Gate | 名称 | G-Lite (T1) | G-Std (T2) | G-Full (T3) |
|------|------|-------------|------------|-------------|
| G1 | 设计完成 | 简要 brief | 标准 brief | 完整 brief + 替代方案 |
| G2 | 设计批准 | 自查 | Role-switch review | 完整 review |
| G3 | 实施完成 | 必需 | 必需 | 必需 |
| G4 | 代码审查通过 | 自查 | Role-switch review | 完整 review |
| G5 | 测试通过 | 聚焦检查 | 标准测试 | 完整测试套件 |
| G6 | 上线部署 | 简单部署 | 分阶段部署 | 完整晋升链 |
| G7 | 监控就绪 | 基本健康 | 标准监控 | 完整监控 |

---

## 5. 设计要求

### G-Lite (T1)
最低要求：目标、非目标、主要风险、验证方法、回滚说明

### G-Std (T2)
必需：问题陈述、方法、范围/非范围、影响文件、约束、成功标准、停止条件、变更类别

### G-Full (T3)
T2 全部加上：现状分析、至少 2 个替代方案、逐方案权衡、监控策略、故障处理策略、发布策略、回滚策略

---

## 6. Review 要求

### G-Lite (T1)
自查：理解正确？更简单方案？隐藏假设？

### G-Std (T2)
Role-switching review: 对齐目标？复杂度适当？失败路径覆盖？可测试？可监控？可发布可回滚？
产出 review 结果：pass / conditional pass / reject

### G-Full (T3)
T2 全部加上：独立 review（调用 Claude Worker 或等效外部审查）
对 Class C/D，显式包含兼容性/回滚/权限影响分析

---

## 7. 实施规则

始终：
- 按批准的设计执行
- 职责清晰
- 关键行为可验证
- 避免静默错误吞噬
- 留够日志供运行时诊断
- 外部调用包含故障处理

禁止：
- 未验证就声称稳定
- 硬编码假结果来通过测试
- 聚焦修复变成无关大重构
- 将推断呈现为已验证事实

---

## 8. 验证规则

### 测试层级

| Level | 范围 |
|-------|------|
| L1: 单元 | 单函数/模块 |
| L2: 集成 | 跨模块、API 契约、DB |
| L3: 系统/E2E | 完整规范用户路径端到端 |
| L4: 验收 | 设计意图满足 |

### 每层级最低要求

| Tier | 必需层级 | Gate 规则 |
|------|----------|----------|
| T1/G-Lite | L1 | 函数级检查即可 |
| T2/G-Std | L1 + L2 + 至少 1 条 L3 路径 | 仅 L1 明确不够 |
| T3/G-Full | L1 + L2 + 完整 L3 + L4 | 包括 E2E 和验收 |

**硬规则**：T2+ 任务中如果 delegate 只提供 L1 证据，controller 必须拒绝或要求补充测试。

### 按项目阶段差异化

以上为分级门槛。按项目阶段（P1-P4）差异化的测试策略、覆盖率目标、G5 检查清单和项目类型特殊要求，详见 `testing-standards.md`。

---

## 9. 发布规则

### G-Lite (T1)
保留旧版本，知道如何回滚，发布后运行主路径。

### G-Std (T2)
变更摘要、回滚说明、止损方法、观察点。分阶段验证后再上线。

### G-Full (T3)
更强的发布清单，谨慎晋升，Class D 不得直接上线。
管道 WIP 限制：晋升中最多 1 个活跃任务。

---

## 10. 监控与事故规则

监控发现变为结构化任务。

三条处理路径：
- **Path A** — 有界可自动处理：已充分理解，无治理变更
- **Path B** — Controller 分诊：需判断
- **Path C** — 事故/热修复：立即稳定，再回归生命周期

关闭条件：不是代码部署时关闭，而是 **3 次连续健康观察周期** 后关闭。
复发：7 天内 3 次 → 自动升级严重度。

---

## 11. 完成纪律

每个重要任务产出证据回答：
- 改了什么？
- 为什么改？
- 如何 review 的？
- 如何验证的？
- 如何回滚？
- 运行时需要观察什么？
- 什么仍然不确定？

禁止将不确定性包装为确定性。
标记未验证项为：未确认、推断、需验证、需人工确认。

---

## 12. 语言纪律

拒绝使用："应该没问题""大概修好了""逻辑上完整了""大概安全"

替代用语：已实施、已验证、未验证、未确认、推断、需手动确认、风险仍存在、未知。

---

## 13. 制品链

每个制品携带：task_id、change_id、parent_artifact、decision_by。
从任何晋升记录都可追溯到原始设计 brief。
